About Cyber Toolkit

Cyber Toolkit is a desktop cybersecurity utility built for Windows. It brings together common network, recon, forensics, and security analysis tools into a single offline-first application — no web server, no cloud dependency, no external accounts required.

Key Facts

— Runs entirely locally as a desktop application

— Built with Python, Flask, and pywebview

— Windows-focused — uses native Windows APIs where possible

— All results cached locally in JSON — no data leaves your machine

— Packagable as a standalone .exe via PyInstaller

How It Works

The app starts a local Flask web server on 127.0.0.1:5000 and launches a native desktop window via pywebview pointed at that server. From your perspective it behaves like a regular desktop app — there is no browser involved and no network exposure. All tool logic runs in Python on your machine and results are rendered back through Jinja2 HTML templates.

Each tab in the sidebar corresponds to a Flask route. When you click a button or submit a form, a standard HTTP POST is made to the local server, the relevant Python tool runs, the result is stored in a central STATE dictionary, and the page re-renders with the new data. Previous scan results are persisted to a local JSON cache so they survive app restarts.

Architecture

Cyber Toolkit follows a simple local client-server architecture entirely contained within a single process on your machine.

Stack

Frontend — Bootstrap 5 HTML/CSS/JS served by Flask's Jinja2 template engine. No JavaScript frameworks. Forms use standard HTTP POST.

Backend — Python Flask application running on 127.0.0.1:5000. All tool logic is organised into modules in the tools/ directory.

Desktop wrapper — pywebview renders the Flask app inside a native OS window using the system's built-in WebView component (Edge WebView2 on Windows). No Electron, no Chromium bundled.

State Management

All tool results are stored in a central STATE dictionary in app.py. This is passed into every template render so any tab can access any result without page-specific state.

Results are also persisted to JSON files via a caching layer (tools/caching_tools.py). On startup, cached results are loaded back into STATE so previous scans are immediately visible without re-running.

│   ├── security_tools.py   ← Vuln scanner, persistence, firewall, privesc

├── templates/              ← Jinja2 HTML templates (one per page)
├── static/
│   ├── style.css
│   └── images/
├── cache/                  ← JSON result cache files (auto-created)
└── quarantine/             ← Malware sandbox quarantine directory
Action Routing Pattern

Every form in the UI contains a hidden action input. The Flask route reads this value and dispatches to the appropriate tool function:

action = request.form.get("action")
if action == "vuln_scan":
    STATE["vuln_results"] = run_vulnerability_scanner(...)
elif action == "persistence_scan":
    STATE["persistence_results"] = persistence_detection()
Packaging

The app can be packaged into a standalone .exe using PyInstaller. A resource_path() helper resolves template and static file paths correctly whether running from source or from a PyInstaller bundle (sys._MEIPASS).

Flask is started with use_reloader=False to prevent PyInstaller's frozen process from launching a second instance on startup.

Data & Privacy

All scan results are stored locally on your machine in the cache/ directory. No telemetry, analytics, or usage data is collected or transmitted.

Some tools make outbound network requests by design — for example, WHOIS lookups, IP geolocation, CVE database queries (OSV/NVD), and DNS resolution. These requests are always initiated explicitly by you clicking a button and involve only the target data you provide.

No credentials, scan results, or system data are ever sent to external servers automatically.

CVE Data Accuracy

Vulnerability scan results are based on keyword matching against the OSV and NVD databases. Results are indicative only — false positives are expected, and the absence of CVEs does not mean software is secure.

Always verify findings manually before taking action based on scan results.

Malware Sandbox

The malware sandbox performs static analysis only — it does not execute uploaded files. Files submitted to the sandbox are quarantined in the local quarantine/ directory. Handle quarantined files with care and do not execute them outside of a controlled environment.

The sandbox is not a substitute for professional malware analysis or enterprise endpoint detection tools.
